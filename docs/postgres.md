# PostgreSQL

PostgreSQL is required for the ExcaliDash V2 runtime. SQLite is no longer used
at runtime and appears only in legacy migration guidance.

PostgreSQL provides production persistence, consistent schema migrations,
concurrent access, and predictable deployments.

## Variables

- `POSTGRES_DB`: database created by the bundled PostgreSQL container.
- `POSTGRES_USER`: PostgreSQL role used by ExcaliDash.
- `POSTGRES_PASSWORD`: role password; generated by `quickstart.sh`.
- `DATABASE_URL`: Prisma runtime connection string.
- `DIRECT_URL`: direct connection string for migration/tooling workflows.

The Compose file builds `DATABASE_URL` and `DIRECT_URL` from the PostgreSQL
variables. For a backend running directly on the host, the database host is
`localhost`; inside Compose it is the `postgres` service.

## Migrations

Development:

```bash
cd backend
npx prisma migrate dev
```

Apply committed migrations:

```bash
cd backend
npx prisma migrate deploy
```

The backend container applies committed migrations at startup by default.

## Backup with `pg_dump`

Create a plain SQL backup:

```bash
docker compose exec -T postgres pg_dump \
  -U excalidash -d excalidash > excalidash.sql
```

## Restore with `psql`

Restore into an empty, verified target database:

```bash
docker compose exec -T postgres psql \
  -U excalidash -d excalidash < excalidash.sql
```

Test restore procedures before relying on them in production.

## Legacy SQLite migration

There is no SQLite runtime fallback. Export drawings/backups from the legacy
installation, start ExcaliDash V2 with PostgreSQL, and import the exported
data through the dashboard. Keep the legacy database backup until users,
collections, drawings, permissions, and snapshots are verified.
